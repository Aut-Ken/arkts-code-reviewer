# Retrieval v1 Golden set

This directory freezes a small, fixture-only contract for deterministic Retrieval
evaluation. It does not use the local knowledge corpus or a production embedding
provider.

## Frozen contents

- `index.json` contains 20 synthetic, human-curated Clauses and reviewed 8-dimensional
  passage vectors. Its `origin` is `golden_fixture`, so production Retrieval rejects it
  unless the explicit test-only opt-in is enabled.
- `manifest.json` contains exactly 36 self-contained cases, `RG-001` through `RG-036`.
  Every Unit includes its complete Target Platform, exact signals, semantic excerpt,
  routing context, parser quality, budget, and expected result contract.
- `baseline.json` is the canonical byte-for-byte evaluator report for the reviewed
  manifest. It is an additional drift tripwire, not a source of expected results.

The case distribution is intentionally fixed:

| Cases | Mode | Contract exercised |
| --- | --- | --- |
| RG-001..RG-018 | exact | canonical APIs and aliases, decorators, components, requested rules, true negatives, applicability, budget, dispatch, parser degradation, and cross-Unit isolation |
| RG-019..RG-030 | hybrid | semantic-only recall, exact/vector fusion, deterministic ties, applicability filtering, budget, zero-vector negative, and one shared query vector for identical cross-Unit query text |
| RG-031..RG-036 | embedding failure | fail-open-to-exact behavior, explicit degraded diagnostics, negative fallback, applicability filtering, and cross-Unit isolation |

There are 18 exact cases, 12 hybrid cases, and 6 embedding-failure cases. Positive and
negative expectations cover requested rules, ordered results, required and forbidden
rules, Dimension coverage, Unit diagnostics, and pack degradation.

## Review rule

`expected_units` in `manifest.json` is written and reviewed by a human from the fixture
contract. It must never be copied from evaluator output or generated by running
Retrieval. Query-text hashes may be mechanically calculated from the explicitly
reviewed Unit inputs, but their vectors remain reviewed fixture values.

When a contract intentionally changes:

1. Review the changed index records and Unit inputs.
2. Write the new expected rule order, required/forbidden sets, coverage, diagnostics,
   and degraded flag before looking at evaluator output.
3. Run the evaluator and inspect every difference. Do not accept output wholesale.
4. Update `baseline.json` only after all differences are explained and approved.
5. Run the fail-closed tests, including provenance and duplicate-key drift checks.

## Metric definitions

- Recall@5 measures required rules found in each Unit's first five results.
- Precision@5 treats `ordered_rule_ids` as the reviewed relevant set. A reviewed true
  negative with both expected and actual results empty contributes `1.0`; an unexpected
  result in a negative case contributes `0.0`.
- MRR uses the first required rule rank and excludes Units without required rules.
- A perfect run requires every exact order, diagnostic-code set, coverage partition,
  degraded flag, and forbidden-rule assertion to pass.

## Commands

```bash
PYTHONPATH=src python tools/evaluate_retrieval_golden.py \
  --manifest tests/golden/retrieval/manifest.json \
  --strict-baseline tests/golden/retrieval/baseline.json \
  --require-perfect

PYTHONPATH=src pytest -q tests/test_retrieval_golden.py
```

The manifest pins the raw index SHA-256, canonical index identity, Feature Routing
configuration fingerprint, Retrieval configuration fingerprint, and Retrieval version.
The loader rejects unknown fields, duplicate JSON keys, unknown rule IDs, unsafe fixture
files, vector-dimension drift, and any provenance mismatch.
